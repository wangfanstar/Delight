算术边界

// 边界检查 · 加减传播 · 逻辑操作对范围的影响

第4章是全书中最具数学味道的一章，也是最容易被低估的一章。 Warren 在这里回答了一个核心问题：当你对一个有界的整数做运算， 结果的范围是什么？ 这听起来像是简单的算术，但在有符号/无符号的边界切换、溢出、位运算混用的环境下， 答案出人意料地微妙。编译器的值范围分析（Value Range Propagation）、 安全检查的边界推导、整型提升规则——都建立在这一章的数学基础上。

§4-1 整数边界检查 — Checking Bounds of Integers
// THE CLASSIC TRICK: 两次比较 → 一次比较 判断 a ≤ x ≤ b 通常需要两次比较（两次分支）。 Warren 给出了一种将其合并为一次无符号比较的技巧： (unsigned)(x - a) ≤ (unsigned)(b - a)。 这在数组边界检查中极为常用——Java JVM 和现代 C 编译器的边界检查消除（BCE） 都依赖类似的变换，它把两次有符号比较压缩成一次无符号比较，减少了一次分支预测的代价。

// 原理 将 x 平移使 a 成为原点：令 x' = x - a， 则 a ≤ x ≤ b 等价于 0 ≤ x' ≤ b-a。 将 x' 解释为无符号数，负数变成很大的正数（>b-a），自然就越界了。 因此一次无符号比较即可完成双重判断。前提：b ≥ a（合法区间）。
C 实现：有符号边界检查
// §4-1 边界检查: a ≤ x ≤ b (有符号整数) // 传统方法: 2次比较 2次分支 int in_range_naive(int x, int a, int b) { return (x >= a) && (x <= b); } // Warren 优化: 1次无符号比较 int in_range(int x, int a, int b) { return (unsigned)(x - a) <= (unsigned)(b - a); } // 注意: 要求 b >= a (合法区间) // x < a 时: x-a 是负数 → 无符号很大 → > b-a ✓ // x > b 时: x-a > b-a → 越界 ✓ // a ≤ x ≤ b: 0 ≤ x-a ≤ b-a ✓ // 数组下标检查 (Java JVM 常用方式): // if ((unsigned)index >= (unsigned)length) throw AIOOBE // → 同时检测了 index<0 和 index>=length // → 减少一次分支, 对分支预测器更友好

§4-1 边界检查的工业级应用场景
Java JVM — 数组访问
每次 array[i] 访问都插入 (unsigned)i >= length 检查。JIT 通过循环不变量分析消除冗余检查（Bounds Check Elimination, BCE），但底层检查正是 Warren 的单次无符号比较。
Rust — 切片边界
Rust 切片的 index()SliceIndex::index 使用等价于 index >= len 的无符号比较 panic，避免了有符号溢出 UB。
GCC/LLVM — 值范围分析
编译器的 VRP（Value Range Propagation）Pass 在 SSA 形式上追踪每个变量的范围区间，利用 §4-1 的数学消除不可达的边界检查，减少代码体积和分支数量。
Linux 内核 — 用户空间指针
access_ok(addr, size) 检查用户空间地址是否合法，内部用 (unsigned long)(addr + size) > TASK_SIZE，与 Warren §4-1 同构，同时检查了地址溢出和越界。
§4-2 加减法的范围传播 — Propagating Bounds through Add/Subtract
// RANGE PROPAGATION: 结果的范围取决于操作数的范围 如果已知 a ≤ x ≤ b 且 c ≤ y ≤ d， 那么 x + y 的范围是什么？直觉告诉我们是 [a+c, b+d]， 这对实数是正确的，但对有界整数有溢出的陷阱。 Warren §4-2 系统地给出了有符号和无符号整数在加法和减法下的范围传播规则， 以及如何检测传播过程中是否发生了溢出——这是编译器优化和静态分析工具的理论基础。

// §4-2 有符号整数加减法的范围传播 // 设 x ∈ [a,b], y ∈ [c,d], 全为有符号32位整数 // ★ 加法: x + y ∈ [a+c, b+d] // 前提: 不发生溢出! // 溢出条件: a+c < INT_MIN 或 b+d > INT_MAX // 检测方法 (Warren): int add_lo = a + c; int add_hi = b + d; bool add_overflow = (c < 0 && add_lo > a) || // 下界负溢出 (d > 0 && add_hi < b); // 上界正溢出 // 若溢出: 结果范围退化为 [INT_MIN, INT_MAX] (最保守估计) // ★ 减法: x - y ∈ [a-d, b-c] // (减去y, y越大结果越小, y越小结果越大) int sub_lo = a - d; int sub_hi = b - c; bool sub_overflow = (d > 0 && sub_lo > a) || // 下界负溢出 (c < 0 && sub_hi < b); // 上界正溢出 // 重要推论: 若 x ∈ [a,b] 且 a,b ≥ 0，则: // x+y ≥ 0 当 y ≥ -a (即 y ≥ -(MIN_OF_X)) // 这是编译器消除溢出检查的关键依据
// §4-2 无符号整数加减法的范围传播 // 设 x ∈ [a,b], y ∈ [c,d], 全为无符号32位整数 // ★ 加法: x + y ∈ [a+c, b+d] (模2³²) // 无符号溢出检测 (Warren §4-2): unsigned add_lo = a + c; unsigned add_hi = b + d; bool lo_wrapped = add_lo < a; // 进位 → 模2³²回绕 bool hi_wrapped = add_hi < b; // 进位 → 模2³²回绕 // 若 lo 回绕但 hi 未回绕: 范围跨越 UINT_MAX // → 保守估计为 [0, UINT_MAX] (全范围) // 若两者都回绕: 结果仍然是连续区间 [a+c, b+d] (均wrap) // 若两者都不回绕: 正常 [a+c, b+d] // ★ 减法: x - y ∈ [a-d, b-c] (模2³²) // 无符号借位检测: unsigned sub_lo = a - d; unsigned sub_hi = b - c; bool sub_wrapped = (a < d) || (b < c); // 若 a < d: a-d 借位 → 模2³² 很大 // 这在 Linux 内核的指针差值计算中必须小心! // 关键应用: 证明 (x & mask) 不溢出 // 因为 AND 永远不增大操作数 → AND 结果永远 ≤ 两个操作数 // → 无符号加法 (x&m) + (y&n) ≤ m + n
// §4-2 溢出检测: 确保范围传播后结果仍有意义 // 这些方法同时也是 §2-13 Overflow Detection 的延伸 // ★ 有符号加法溢出检测 (3种等价方法): // 方法1: 检查结果符号 (Warren §2-13, 最经典) int s = a + b; bool ov1 = (~(a ^ b) & (a ^ s)) < 0; // 原理: 只有当 a,b 同号且结果异号时才溢出 // ~(a^b): a,b 同号时最高位为1 // (a^s): a,结果异号时最高位为1 // AND 后最高位 = 两者都满足 = 溢出 // 方法2: C++20 标准库 (最推荐) bool ov2 = __builtin_add_overflow(a, b, &s); // 方法3: 扩展到64位计算后检查 long long s64 = (long long)a + (long long)b; bool ov3 = s64 > INT_MAX || s64 < INT_MIN; // 方法4: 利用 LLVM/GCC 产生的 ADDO 指令 // x86: ADD + INTO/JO (overflow flag) // ARM: ADDS + BVS (V flag) // ★ 无符号加法溢出 (进位检测): unsigned u = a + b; bool uov = u < a; // 若进位则 u < a ✓ (1条指令) // ★ Rust: checked_add / saturating_add / wrapping_add // → 把溢出行为显式化, 彻底消除 UB

§4-3 逻辑操作的范围传播 — Propagating Bounds through Logical Ops
// THE HARDER PROBLEM: 位运算不保持线性结构 加法的范围传播是线性的（区间加区间还是区间），但 AND/OR/XOR 的范围传播 不遵守线性规则。例如：x ∈ [3, 5]（即 011 ~ 101）， y ∈ [0, 15]（即 0000 ~ 1111），则 x & y 的上界是多少？ 直觉说是 min(5,15) = 5，但实际上需要逐位分析。 Warren §4-3 给出了基于"位掩码"的上下界计算方法——这是编译器做 位运算结果范围推断的理论根基，也是 eBPF 验证器的核心算法之一。
AND 的范围传播
// §4-3 AND 操作: z = x & y // x ∈ [a,b], y ∈ [c,d] (无符号) // 上界: z_max = min(b, d) // AND 永远不增大操作数，所以结果 ≤ 两者中的较小者 unsigned z_max = min(b, d); // 下界: 更复杂，Warren 给出以下方法: // 对于无符号: z_min = 0 (保守估计，总是正确的) // 精确下界需要逐位分析 (见 §4-3 详细推导) unsigned z_min_conservative = 0; // Warren 精确算法 (基于区间的位传播): // 令 m = ~(b & d): 找出 b 和 d 中最高的不同位 unsigned m = ~(b & d); // m 的最高位标识了需要"退一步"的位置 // 这个算法的完整版见Warren原书图4-2 // 关键性质: // (x & C) ∈ [0, C] — AND 常数时上界就是常数 // (x & x) = x — 平凡情况 // (x & 0) = 0 — AND零得零 // eBPF 验证器 (Linux 5.x) 的 AND 范围推断: // dst_reg.umax_value = min(dst_reg.umax, src_reg.umax) // dst_reg.umin_value = 0 (保守)
OR / XOR 的范围传播
// §4-3 OR 操作: z = x | y // x ∈ [a,b], y ∈ [c,d] (无符号) // 下界: z_min = max(b, d) (OR只会置位, 不清位) // 更精确: z_min = max(a, c) (但 max(b,d) 不对!) // 实际: z_min = a | c (两者的"已知1位"都保留) unsigned or_min = a | c; // 下界: 两者都是1的位必然是1 // 上界: Warren §4-3 精确算法 // 令 m = 找到 b|d 中使区间"向上扩展"的最高位 unsigned m_or = b | d; // 保守上界: 两者可置位的所有位都置1 unsigned or_max = b | d; // 简单保守估计 // XOR 操作: z = x ^ y // XOR 的范围传播最难精确, 通常用保守估计: // z_min = 0 (两者相同时XOR=0) // z_max = b | d (所有可能置位的位) unsigned xor_min = 0; unsigned xor_max = b | d; // 关键性质: // (x | C) ∈ [C, C | UINT_MAX] — OR 常数时下界是常数 // (x | 0) = x — OR零不变 // (x ^ x) = 0 — XOR自身得零 // (x ^ ~x) = UINT_MAX // LLVM ConstantRange::binaryOr / binaryAnd // 实现了 Warren §4-3 的完整算法
📦 真实案例：eBPF 验证器的位范围分析
// LINUX KERNEL eBPF VERIFIER — kernel/bpf/verifier.c Linux eBPF 程序在加载时必须通过内核验证器的静态分析， 确保每条指令不会产生越界内存访问、非法指针运算等危险操作。 验证器对每个寄存器维护一个"值范围"结构体（struct bpf_reg_state）， 包含 smin/smax（有符号）和 umin/umax（无符号）四个边界， 在每条 AND/OR/ADD/SUB 指令后用 §4-2/§4-3 的算法更新这些边界。 如果验证器无法证明某次内存访问是安全的，程序加载失败。
// 简化版 eBPF 寄存器状态 (参考 linux/bpf.h) struct bpf_reg_state { s64 smin_value; // 有符号下界 s64 smax_value; // 有符号上界 u64 umin_value; // 无符号下界 u64 umax_value; // 无符号上界 u64 var_off.mask; // 未知位的掩码 (tnum) u64 var_off.value;// 已知位的值 (tnum) }; // AND 指令后更新 (scalar_min_max_and): void scalar_min_max_and(reg1, reg2, dst) { dst.umax_value = min(reg1.umax_value, reg2.umax_value); // Warren §4-3 dst.umin_value = 0; // 保守下界 // 若两者都非负: 有符号边界同无符号 if (reg1.smin_value >= 0 && reg2.smin_value >= 0) { dst.smax_value = dst.umax_value; dst.smin_value = 0; } } // 典型验证场景: // r1 = map_value_ptr (已知指向合法内存) // r2 = user_input (未知, smin=-∞, smax=+∞) // r2 &= 0xFF → 验证器更新: r2 ∈ [0, 255] (§4-3 AND常数) // r1 += r2 → 验证器更新: r1偏移 ∈ [0, 255] (§4-2 加法传播) // 读 [r1] → 偏移已证明在界内, 允许! ✓

§4-4 溢出检测与饱和算术 — Overflow Detection & Saturating Arithmetic
// OVERFLOW: THE ROOT OF ALL EVIL IN INTEGER ARITHMETIC §4-2 的范围传播告诉我们结果范围，但当实际计算发生时，溢出可能已经造成了破坏。 Warren §4-4 在传播理论之上给出了运行时溢出检测的高效实现， 以及一种优雅的替代：饱和算术（Saturating Arithmetic）—— 溢出时把结果夹在 [INT_MIN, INT_MAX]，而不是回绕。 这在图像/音频/DSP处理中极为重要（像素值溢出不应该从 255 回绕到 0）。

饱和加法实现 (多种方法)
// §4-4 饱和加法: 溢出时夹到边界 // 方法1: 朴素方法 (有分支, 最慢) int sadd_naive(int a, int b) { long long r = (long long)a + b; if (r > INT_MAX) return INT_MAX; if (r < INT_MIN) return INT_MIN; return (int)r; } // 方法2: Warren §4-4 无分支 (位操作) int sadd_branchless(int a, int b) { unsigned ua = a, ub = b; unsigned s = ua + ub; // 溢出检测: 同号相加且结果异号 unsigned ov = (~(ua ^ ub) & (ua ^ s)) >> 31; // ov=1 → 溢出; ov=0 → 正常 // 构造饱和边界值: // a<0 → sat = INT_MIN = 0x80000000 // a>=0 → sat = INT_MAX = 0x7FFFFFFF unsigned sat = (ua >> 31) + (unsigned)INT_MAX; return (int)(ov ? sat : s); // 完全无分支: SELECT(ov, sat, s) // 编译器会生成 CMOV (条件移动) 指令 } // 方法3: GCC/Clang 内建 (推荐) int sadd_builtin(int a, int b) { int s; if (__builtin_add_overflow(a, b, &s)) return a > 0 ? INT_MAX : INT_MIN; return s; } // 方法4: x86 SIMD 饱和加法 (最快) // _mm_adds_epi16 — SSE2 饱和加法 (8个16位) // _mm_adds_epu8 — SSE2 无符号饱和加法 (16个8位) // → 音频/图像处理的标准选择
饱和减法 / 无符号饱和算术
// §4-4 无符号饱和加法 unsigned usadd(unsigned a, unsigned b) { unsigned s = a + b; // 无符号加法溢出检测: s < a (进位) return s | -(s < a); // s < a 时: -(1) = 0xFFFFFFFF → s|0xFFFF = UINT_MAX // s >= a 时: -(0) = 0 → s|0 = s // 巧妙: 1条减法+1条OR, 完全无分支! } // 无符号饱和减法 (下界为0) unsigned ussub(unsigned a, unsigned b) { unsigned d = a - b; // a < b 时: d借位 → 很大, 应返回0 return d & -(a >= b); // a >= b: -(1) = 0xFFFF → d & 0xFFFF = d ✓ // a < b: -(0) = 0 → d & 0 = 0 ✓ } // ARM 架构原生支持: // QADD — 有符号饱和加 (Q flag) // QSUB — 有符号饱和减 // UQADD — 无符号饱和加 // UQSUB — 无符号饱和减 // → DSP/音频处理器标配, 1周期完成 // 应用: 音频混音 (16位样本) // mixed = sadd16(track1, track2) // 避免: 255+255=510 回绕到 254 (爆音!) // 饱和: 255+255=255 (正确截幅)

§4-5 历史背景与跨语言对比
🕰 Ada (1983)
Ada 是最早将整数子范围类型内置到语言规范的语言。 subtype Index is Integer range 0..99; 编译器自动在边界处插入检查，越界抛出 Constraint_Error。 Warren §4-1 的"单次无符号比较"正是此类运行时检查的高效实现。 Ada 至今仍是航空航天软件（DO-178C）的首选语言，其整数语义被精确定义到比特级别。
🦀 Rust (2015)
Rust 的整数类型在 debug 模式下默认检测溢出（panic）， release 模式下回绕（与 C 无符号行为一致）。 标准库提供四族方法： checked_add（返回 Option）， saturating_add（§4-4 饱和语义）， wrapping_add（显式回绕）， overflowing_add（返回 (结果, 是否溢出)）。 彻底消除了 C/C++ 中有符号溢出 UB 的安全隐患。
☕ Java (1995)
Java 整数始终回绕（两种补码，无 UB）， 但数组越界抛 ArrayIndexOutOfBoundsException。 JVM 的 JIT（HotSpot C2）用 Warren §4-1 的无符号单次比较实现数组越界检查， 并通过循环无关量提升（Loop Invariant Code Motion）和 边界检查消除（BCE）将热循环中的检查从 O(n) 降到 O(1)。 JDK 16+ 加入了 Math.clamp() 实现饱和语义。
🛡 C (UB 地雷区)
C 标准规定有符号整数溢出是未定义行为（UB）， 这允许编译器在 -O2 下删除溢出检查代码（认为溢出不会发生）。 著名案例：Linux 内核曾因 i+1 < i 检查被 GCC 优化删除 导致安全漏洞（CVE-2009-1897）。 -fsanitize=undefined（UBSan）可在运行时检测， 但生产环境开销较大。Warren §4-4 的无分支位操作方案是在 C 中安全处理溢出的最优解。
🔥 Swift (2014)
Swift 默认对所有整数运算进行溢出检查，溢出时 trap（崩溃）而非 UB。 提供溢出操作符 &+、&-、&* 用于显式允许回绕（性能敏感路径）。 Swift 的整数溢出语义经过 ABI 稳定化（Swift 5）， 底层实现完全依赖 LLVM 的 @llvm.sadd.with.overflow 内建函数， 对应 x86 的 ADD + JO 双指令，性能接近无检查版本。
🔧 DSP / 嵌入式
数字信号处理器（DSP）将饱和算术作为硬件原生特性。 TI C55x、Qualcomm Hexagon、ARM Cortex-M 的 DSP 扩展 均提供单周期饱和加减乘指令。 音频编解码（MP3/AAC/Opus）、图像滤波（卷积核运算）和 神经网络量化（INT8 推理）大量依赖饱和语义， 避免回绕导致的信号失真和模型精度损失。

§CRIT 深度点评 — 第4章的价值与局限

第4章：被低估的理论基石

✦ 数学严谨性远超同类书籍

Warren 在本章给出了完整的区间算术（Interval Arithmetic）框架， 区别于一般的"经验技巧"，这是有数学证明支撑的系统性理论。 每一个传播规则都附有溢出条件的精确描述，这在同类位操作书籍中极为罕见。 LLVM 的 ConstantRange 类（约 3000 行 C++）和 GCC 的 VRP Pass 本质上就是对这一框架的工程实现。

✦ §4-1 的"单次无符号比较"影响深远

这个看似简单的技巧渗透到了 JVM 字节码验证器、eBPF 安全检查、 Rust 切片索引、WebAssembly 线性内存访问等几乎所有现代运行时的安全基础设施中。 本质是将"有界性判断"从代数域转换到模算术域， 利用无符号整数的"负数即大数"特性一次完成双重检查。

✦ 逻辑操作传播的精确性有所保守

§4-3 对 XOR 的处理采用了保守估计（下界为0，上界为 b|d）， 这在编译器优化中会导致精度损失。现代编译器（LLVM 14+）已引入 tnum（tristate number）表示，用三态位（0/1/unknown） 追踪每一位的已知状态，精度远超区间方法。 Warren 的区间方法更适合理解原理，tnum 更适合工程实践。

✦ 饱和算术一节是 DSP 工程师的必读内容

§4-4 的无分支饱和加法 s | -(s < a) 是教科书级别的技巧—— 利用无符号溢出检测（s<a）生成全1或全0掩码，与结果 OR 实现饱和。 整个操作在现代 CPU 上只需 2-3 条指令，比 if-else 分支快约 3 倍。 这对实时音频（44.1kHz × 32声道的混音引擎）的性能至关重要。

⚠ 时代局限：未覆盖 C/C++ UB 语义

第一版写于 2002 年，彼时编译器对有符号溢出 UB 的激进优化尚不普遍。 现代 GCC/Clang 在 -O2 下可能删除基于溢出假设的代码， 使部分示例在实际编译时行为不符合预期。 建议配合 -fwrapv（GCC）或 -fsanitize=undefined 使用， 或改写为无符号操作（UB-free）。这是本章最大的工程陷阱。
