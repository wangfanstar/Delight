📐 第一组：内存对齐场景 Q1–Q5
Q1 OS内核
Linux内核需要将任意地址 addr 向上对齐到4KB页面边界。内核宏 PAGE_ALIGN 展开后的本质是什么？
PAGE_SIZE = 4096 = 2¹²，要求结果 ≥ addr 且是 4096 的倍数
基础
Aaddr & 0xFFFFF000 — 直接AND掩码向下取整
B(addr + 4095) & (~4095) — 加垫脚石后AND掩码
C(addr / 4096 + 1) * 4096 — 整除后乘回来
Daddr | 4095 — OR低位全1
✅ 正确答案：B

公式：(x + d - 1) & (-d)，当 d=4096 时展开为 (addr + 4095) & 0xFFFFF000。

原理：+4095 保证不足一页的部分"进位"到下一页，& (~4095) 清除低12位（等价于 & (-4096)，因为 -4096 的二进制正好是低12位全0）。

A 是向下对齐（去尾），不是向上。C 有除法（慢，且addr已对齐时会错误+1）。D 的 OR 是置位低位，完全相反。

Q2 性能优化
某高性能 Ring Buffer 的大小必须是2的幂。下面代码使用了取模运算，现在要把它替换为等价但更快的位运算形式，正确的替换是？
基础 
// size = 1024（2的幂），index 递增  
index = (index + 1) % size;  // 慢：除法指令
Aindex = (index + 1) & ~size
Bindex = (index + 1) & (size - 1)
Cindex = (index + 1) | size
Dindex = (index + 1) >> 10
✅ 正确答案：B

当 size 是2的幂时，x % size 等价于 x & (size - 1)。

size=1024=0x400，size-1=0x3FF=二进制低10位全1，AND操作保留低10位 = 取模。

性能差距：% 需要 DIV 指令（20~40周期），& 是1个周期。Linux kfifo、Disruptor 都用这个技巧。

A 的 ~size 是错误掩码；C 的 OR 会置位而非取模；D 的右移是求商不是余数。

Q3 SIMD优化
你正在编写 AVX2 SIMD 代码，需要在 assert 中检查一个指针 ptr 是否满足32字节对齐（AVX2要求）。最简洁的检测表达式是？
基础
A((uintptr_t)ptr % 32) == 0
B((uintptr_t)ptr & 31) == 0
C((uintptr_t)ptr & 32) == 0
D((uintptr_t)ptr >> 5) << 5 == (uintptr_t)ptr
✅ 正确答案：B

对齐检测公式：(x & (d-1)) == 0，d=32，d-1=31=0x1F=低5位全1。

AND后低5位若全为0，说明地址的低5位本来就是0，即32字节对齐。

C 是错误的：& 32 只检查第5位是否为0（即检测64字节对齐的一个必要但不充分条件）。A 正确但用了除法。D 正确但冗余（移位组合）。

Q4 Linux内核
Linux内核 include/linux/mm.h 中有宏 __ALIGN_KERNEL_MASK(x, mask)，其实现展开后等价于下面哪个表达式？（mask = d - 1）
进阶 
// 已知：mask = d - 1，d 是2的幂  
// __ALIGN_KERNEL_MASK(x, mask) = ???
Ax & ~mask — 直接AND反码
B(x + mask) & ~mask — 加mask后AND反码
Cx | mask — OR低位全1
D(x - 1) & ~mask — 减1后AND反码
✅ 正确答案：B

Linux内核实现：(x + mask) & ~mask，其中 mask = d-1。

代入展开：(x + (d-1)) & ~(d-1) = (x + d - 1) & (-d)，正是 Warren 书中的向上对齐公式。

A 是向下对齐（去掉低位）。这正是 ALIGN_DOWN 的实现。B 才是向上对齐 ALIGN_UP / PAGE_ALIGN。

Q5 内存分配
一个内存分配器需要返回16字节对齐的地址（SIMD基础要求）。用户请求分配 size 字节，内部实际分配空间需要向上对齐到16字节的倍数。size = 100 时，实际分配多少字节？
进阶 
// 向上对齐到16字节  
actual = (size + 15) & ~15;  
// size = 100, 求 actual = ?
A100 — 恰好整除，不需要扩充
B112 — 向上到下一个16的倍数
C108 — 100 + 8 字节
D128 — 向上到下一个2的幂
✅ 正确答案：B（112）

计算过程：(100 + 15) & ~15 = 115 & 0xFFFFFFF0

115 = 0x73 = 0111 0011，~15 = 0xFFFFFFF0 = ...1111 0000

AND 后 = 0x70 = 112。

100 / 16 = 6 余 4，所以向上到 7 × 16 = 112。+15再AND的公式自动处理了"余数"到"补足量"的转换。

⬆️ 第二组：flp2 / clp2 场景 Q6–Q10
Q6 Java HashMap
Java HashMap 内部调用 tableSizeFor(cap) 来确定哈希表容量。当用户传入 cap = 100 时，实际分配的桶数组大小是多少？
HashMap 必须保证容量是2的幂，以便用 & 代替 % 做哈希映射
基础 
// JDK 8 tableSizeFor 简化版（就是 clp2）  
int n = cap - 1;  
n |= n >>> 1;  n |= n >>> 2;  
n |= n >>> 4;  n |= n >>> 8;  
n |= n >>> 16;  
return (n < 0) ? 1 : n + 1;  
// cap = 100, 返回值 = ?
A100 — 正好满足需求，不需要扩容
B112 — 向上对齐到16的倍数
C128 — 不小于100的最小2次幂
D256 — 向上到下一个更大的2次幂
✅ 正确答案：C（128）

clp2(100) = 不小于100的最小2次幂 = 128（2⁷）。

计算：n = 99 = 0x63，5步OR-Shift后 n = 0x7F = 127，返回 127+1 = 128。

Josh Bloch 在 JDK 的实现本质上就是 Warren 书里的 clp2 算法，先减1是为了让"正好是2次幂"的输入保持不变（clp2(64) = 64，不会变成128）。

A 错：HashMap 容量必须是2次幂。B 错：112不是2次幂。D 错：256 > 128，clp2 取的是最小的满足条件的值。

Q7 Linux内核
Linux 伙伴系统（Buddy System）分配内存时，用户请求分配 200 个连续页面。伙伴系统只能分配 2 的幂次个页面，它会分配多少页？对应 order 是多少？
order k 表示分配 2ᵏ 个连续页面，页面大小 4KB
进阶 
// 伙伴系统核心：求 order = ⌈log₂(n)⌉  
int order = 0;  
unsigned n = clp2(200);  // 向上取2次幂  
while (n > 1) { n >>= 1; order++; }  
// 分配 2^order 页，order = ?
Aorder=7，分配 128 页 (512KB)
Border=8，分配 256 页 (1MB)
Corder=7，分配 200 页（不浪费）
Dorder=9，分配 512 页 (2MB)
✅ 正确答案：B（order=8，256页）

clp2(200) = 256 = 2⁸，因此 order = 8，分配 256 页 = 1MB。

200 介于 128（2⁷）和 256（2⁸）之间，伙伴系统必须向上取整到 256，会浪费 56 页。

这就是伙伴系统的内碎片（internal fragmentation）——分配的比请求的多。clp2 在此处是核心操作。

A 错：128 < 200，装不下。C 错：伙伴系统只能按2次幂分配，不能精确到200。

Q8 算法设计
FFT（快速傅里叶变换）要求输入数组长度必须是2的幂。你有一个长度为 n = 500 的信号，需要在末尾补零到最近的2次幂长度再做FFT。补零后长度是？
基础 
// 补零到 clp2(n)  
int fft_len = clp2(n);  // n = 500  
memset(arr + n, 0, (fft_len - n) * sizeof(float));  
// fft_len = ?  补了多少个零 = ?
Afft_len=512，补 12 个零
Bfft_len=512，补 0 个零（500不需要补）
Cfft_len=1024，补 524 个零
Dfft_len=500，不需要是2次幂
✅ 正确答案：A（fft_len=512，补12个零）

clp2(500) = 512（2⁹），因为 256 < 500 ≤ 512。需要补 512 - 500 = 12 个零。

计算：n-1=499=0x1F3，OR-Shift传播后=0x1FF=511，+1=512。

FFT/DFT 场景是 clp2 最经典的工程应用。补零（zero-padding）不仅使FFT可行，还能提高频域分辨率。

Q9 性能分析
下面的代码用来找不超过 x 的最大2次幂，但其中有一处 Bug。请找出并选出正确版本。
进阶 
// 有Bug的版本：  
unsigned flp2_bug(unsigned x) {  
    x |= x >> 1;  x |= x >> 2;  
    x |= x >> 4;  x |= x >> 8;  
    x |= x >> 16;  
    return x >> 1;  // ← Bug在这里？  
}
A没有Bug，x >> 1 和 x - (x >> 1) 结果相同
B有Bug：x >> 1 对非2次幂的掩码会错误地移动最高位，应改为 x - (x >> 1)
C有Bug：少了一步 x |= x >> 32，应加上这一行
D有Bug：应该先做 x -= 1 再传播
✅ 正确答案：B

5步OR-Shift传播后，x 变成 0...01...1 形式（从最高1-bit到bit0全为1），例如 x=5 传播后得 0x00000007。

此时 x >> 1 = 0x00000003 = 3，不是2次幂，也不等于 flp2(5) = 4！

x - (x >> 1) = 7 - 3 = 4 ✅。公式的含义是：全1掩码 减去 自身右移一位 = 只剩最高位。

A 错：反例 x=5，flp2(5)=4，bug版返回3。C 错：32位无符号不需要 x>>32（C标准下位移量≥位宽是未定义行为）。D 错：先减1是 clp2 的技巧，不是 flp2。

Q10 溢出陷阱
你的同事写了如下代码为一个大型数组分配内存，在生产环境中偶发崩溃。请指出根本原因。
困难 
// 同事的代码  
void* alloc_pow2(size_t n) {  
    size_t cap = clp2(n);   // 向上取2次幂  
    return malloc(cap);     // 偶发返回NULL  
}  
// 调用方：alloc_pow2(0x80000010UL)  (32位系统)
Amalloc 系统内存不足，是正常的内存压力问题
Bclp2 溢出：输入超过 2³¹ 时，32位无符号 clp2 结果为 0，导致 malloc(0) 返回极小内存
Cclp2 的5步OR-Shift不足以处理超大数，需要额外步骤
Dmalloc 不支持2次幂大小的分配，需要用 aligned_alloc
✅ 正确答案：B（clp2 溢出）

输入 0x80000010 = 2³¹ + 16，其 clp2 应为 2³²，但 32位无符号无法表示 2³²，结果回绕为 0。

于是 malloc(0) 被调用，行为取决于实现（可能返回非NULL小指针），之后写入数据必然越界崩溃。

修复：加溢出检测 if (cap == 0 || cap < n) return NULL;

Warren 书中专门警告：clp2(x) 对于 x > 2³¹（32位）会溢出返回0。这是生产代码中的经典隐患。

🔍 第三组：边界跨越检测场景 Q11–Q15
Q11 DMA传输
DMA 控制器要求单次传输不得跨越4KB页面边界。现在要从地址 0x1FF8 传输 16 字节，这次传输是否需要拆分为两次？
基础 
// 4KB = 2¹², 检测是否跨越4KB边界  
int crosses = ((a ^ (a + l - 1)) >> 12) != 0;  
// a = 0x1FF8, l = 16  
// a + l - 1 = 0x1FF8 + 15 = 0x2007
A不需要拆分，0x1FF8 和 0x2007 在同一个4KB页内
B需要拆分，0x1FF8 在第0页，0x2007 在第1页，跨越了4KB边界
C无法判断，需要知道物理地址映射
D需要拆分，因为传输长度16不是4KB的约数
✅ 正确答案：B（需要拆分）

XOR检测：0x1FF8 ^ 0x2007 = 0x3FFF，右移12位 = 0x3 ≠ 0，跨越了边界。

直观理解：0x1FF8 的高位是 0x1（第0页），0x2007 的高位是 0x2（第1页），高位不同 → 跨页。

实际需要拆分为：第一次 0x1FF8 传 8 字节到 0x1FFF，第二次 0x2000 传 8 字节到 0x2007。

页内偏移公式验证：0x1FF8 & 0xFFF = 0xFF8 = 4088，4088 + 16 = 4104 > 4096 → 跨页 ✅

Q12 DMA传输
同样是DMA检测，但这次传输长度 l = 0（用户传入空传输请求）。直接套用 XOR 公式 ((a ^ (a+l-1)) >> 12) != 0 会发生什么？
进阶 
// 当 l = 0 时：  
// a + l - 1 = a + 0 - 1 = a - 1  
// 若 a = 0x1000, a - 1 = 0x0FFF (正常)  
// 若 a = 0x0000, a - 1 = ???   (无符号)
A没有问题，l=0 表示不传输，XOR结果自然为0
B当 a=0 时 a-1 无符号下溢为 0xFFFFFFFF，XOR后误报"跨越"，需在函数入口特判 l==0
Cl=0 时公式退化为 (a ^ a) >> 12 = 0，没有问题
D编译器会自动优化掉 l=0 的分支
✅ 正确答案：B

l=0 时，a + l - 1 = a - 1。若 a=0（无符号），0 - 1 = 0xFFFFFFFF（下溢）。

0x0000 ^ 0xFFFFFFFF = 0xFFFFFFFF，右移12位 = 0xFFFFF ≠ 0，误报跨越！

Warren 书中明确指出：公式要求 l ≥ 1。Linux DMA 驱动在调用跨越检测前都有 if (!len) return; 保护。

C 是错的：l=0 时 a+l-1 = a-1 ≠ a，不是 a ^ a。

Q13 缓存优化
你正在优化一个热点结构体，要检测一个64字节的结构体（从地址 addr 开始）是否跨越了缓存行边界（缓存行=64字节=2⁶）。检测公式中 k 应该是多少？
基础 
// 检测结构体是否跨越缓存行  
int spans_cacheline =  
    ((addr ^ (addr + 63)) >> k) != 0;  
// 缓存行 = 64B = 2ᵏ, k = ?
Ak = 64，因为检测64字节对象
Bk = 7，因为 2⁷ = 128
Ck = 6，因为缓存行 = 64 = 2⁶
Dk = 8，向上取到下一个2次幂
✅ 正确答案：C（k=6）

公式 ((a ^ (a+l-1)) >> k) != 0 中，k = log₂(边界大小)。

缓存行 = 64 = 2⁶，所以 k = 6。右移6位后若高位不同，则跨越了64字节边界。

规律：检测 2ᵏ 字节边界跨越，右移量就是 k。4KB页 k=12，64B缓存行 k=6，256B DRAM行 k=8。

k 是边界大小的 log₂，不是对象的大小！对象大小决定 l（即 a+l-1 中的 l）。

Q14 虚拟内存
内核的 IOMMU 驱动需要检测一段 DMA 传输 a=0x3000, l=0x1001 是否跨越4KB边界。下列哪个等价表达式与 XOR 公式完全等价且更易理解？
4KB = 0x1000，页内偏移 = a & 0xFFF
进阶 
// XOR 方法（书中标准）:  
int cross = ((a ^ (a + l - 1)) >> 12) != 0;  
// a=0x3000, l=0x1001  
// 等价写法是？
A(a & 0xFFF) + l > 0x1000 — 页内偏移加长度超过页大小
Ba % 4096 + l > 4096 — 取模后相加
C(a + l) > (a & ~0xFFF) + 0x1000 — 终点超过本页末尾
DA 和 C 均正确，且结果与 XOR 方法等价
✅ 正确答案：D（A 和 C 均正确）

验证 a=0x3000, l=0x1001：

A: (0x3000 & 0xFFF) + 0x1001 = 0 + 0x1001 = 0x1001 > 0x1000 → 跨越 ✅

C: 0x3000 + 0x1001 = 0x4001 > 0x3000 + 0x1000 = 0x4000 → 跨越 ✅

XOR: 0x3000 ^ 0x4000 = 0x7000, >>12 = 7 ≠ 0 → 跨越 ✅

三种写法在 l ≥ 1 时完全等价。A 最直观（页内偏移视角），C 最像"终点越界"检测，XOR 方法最简洁（无分支，1~2条指令）。

Q15 IBM System/370
Warren 书中引用了 IBM System/370 程序员 Stephenson 的方案：用 ¬(a | -d) < l 检测跨越2ᵏ边界（d=2ᵏ）。当 a=0x0FF8, l=16, d=0x1000 时，该表达式的计算结果是什么？
困难 
// Stephenson [CJS] 方案  
// ~(a | -d) < l  
// a=0x0FF8, l=16, d=0x1000 (4KB)  
// -d = -0x1000 = 0xFFFFF000 (32位无符号)  
// a | -d = 0x0FF8 | 0xFFFFF000 = ?  
// ~(...) = ?   < 16 ?
A结果为 false（不跨越），因为 0x0FF8+16=0x1008 在同一页内
B结果为 true（跨越），~(a|-d)=7 < 16，跨越4KB边界
B结果为 false，~(a|-d) = 0x7 ≥ 16 不成立，所以不跨越
D公式在此处无法判断，需要用 XOR 方法
✅ 正确答案：B（跨越，~(a|-d)=7 < 16）

逐步计算：

① -d = -0x1000 = 0xFFFFF000（32位二补码）

② a | -d = 0x0FF8 | 0xFFFFF000 = 0xFFFFFFF8

③ ~0xFFFFFFF8 = 0x00000007 = 7

④ 7 < 16 → true，跨越边界

公式含义：~(a|-d) = 本块内 a 之后还剩余的字节数 - 1（即到页边界的距离-1）。若剩余空间 < l，则跨越。

验证：0x0FF8 的页内偏移 = 0xFF8 = 4088，到页尾还有 4096-4088=8 字节，但 l=16 > 8 → 跨越 ✅

🧩 第四组：综合应用与代码阅读 Q16–Q20
Q16 代码阅读
阅读下面这段内存池初始化代码，指出它存在什么问题，并选出正确的修复方案。
困难 
// 内存池：分配 n 字节，容量向上取2次幂  
void* pool_init(size_t n) {  
    unsigned cap = n - 1;       // step1  
    cap |= cap >> 1;  
    cap |= cap >> 2;  
    cap |= cap >> 4;  
    cap |= cap >> 8;  
    cap |= cap >> 16;  
    cap = cap + 1;              // step2: clp2  
    return malloc(cap);  
}  
// 问题：pool_init(0) 和 pool_init(0x80000001UL) 各会怎样？
A没有问题，clp2 实现完全正确
Bn=0 时 step1 下溢为 0xFFFFFFFF，导致 cap 溢出为 0，malloc(0) 被调用
Cn=0x80000001 时 clp2 结果为 0（溢出），malloc(0) 被调用；n=0 也有下溢问题；两者都需修复
DOR-Shift 步骤不够，64位系统需要额外的 cap |= cap >> 32
✅ 正确答案：C（两个 Bug 都存在）

Bug 1：n=0：n-1 = 0-1 = 0xFFFFFFFF（无符号下溢），5步传播后 cap 仍为 0xFFFFFFFF，+1 后 cap=0，malloc(0) 被调用。

Bug 2：n=0x80000001：clp2 应为 2³²，但 32位无符号表示不了 2³²，cap 溢出回绕为 0，malloc(0) 再次被调用。

正确修复：

if (n == 0) return NULL;
if (n > 0x80000000U) return NULL; // 或用 size_t/64位

Warren 书中专门指出这两个边界情况，生产代码必须同时处理。

Q17 性能优化
一个网络包处理系统需要将每个包的起始地址向上对齐到32字节（避免 false sharing）。以下四种写法，哪些在功能上完全等价且结果相同？
进阶 
// addr 是 uintptr_t，d=32  
A: (addr + 31) & ~31  
B: (addr + 31) & -32  
C: (addr + 32 - 1) & ~(32 - 1)  
D: addr + (32 - addr%32) % 32
A只有 A 正确，其他三种有误
BA、B、C 等价，D 在 addr 已对齐时结果不同（会加32）
CA、B、C、D 四种完全等价
DA、B、C 等价，D 在 addr%32==0 时正确，但有多余的除法开销
✅ 正确答案：C（四种完全等价）

A = B：~31 = 0xFFFFFFE0 = -32（二补码），等价 ✅

A = C：展开 (addr+32-1)&~(32-1) = (addr+31)&~31 ✅

D 的分析：当 addr%32=0 时，(32-0)%32=0，addr+0=addr ✅；当 addr%32=k>0 时，(32-k)%32=32-k，addr+32-k = 下一个32倍数 ✅

D 功能正确，但含有两次除法（% 运算），性能远差于 A/B/C 的纯位运算。这正是为什么工程上首选 (x+d-1)&(-d) 的原因。

Q18 舍入应用
Warren 书中有一道习题：将无符号整数 x 舍入到最近的8的倍数（半程向上）。给定 x = 100，正确的舍入结果和公式是？
8的倍数：0,8,16,...,96,104,... 半程=4，100离104更近
进阶
A结果=96，公式 x & ~7（向下取整）
B结果=104，公式 (x + 4) & ~7（半程向上舍入）
C结果=104，公式 (x + 7) & ~7（向上取整）
D结果=100，100已经是最近的8倍数的舍入值
✅ 正确答案：B（结果104，公式 (x+4)&~7）

100 mod 8 = 4，恰好是半程，半程向上规则 → 进位到 104。

公式 (x+4)&~7：100+4=104，104&~7=104 ✅

对比三个公式的作用：

x & ~7 = 向下取整（96）

(x+4) & ~7 = 最近舍入，半程向上（书中习题1(a)答案）

(x+7) & ~7 = 向上取整（104，但对x=96也会得到104，多算了一步）

C 是向上对齐，不是"最近舍入"——若 x=97，C 给104但最近值是96；B 给96 ✅

Q19 操作系统
Linux 伙伴系统中，已知页帧号 pfn=0x0C（十六进制12），order=2（每块4页）。其"伙伴"块的页帧号是多少？
伙伴公式：buddy_pfn = pfn ^ (1 << order)
进阶 
// Linux buddy_pfn 计算  
buddy_pfn = pfn ^ (1 << order);  
// pfn = 0x0C = 1100b, order = 2  
// buddy_pfn = ?
A0x08（十进制8）
B0x10（十进制16）
C0x0E（十进制14）
D0x04（十进制4）
✅ 正确答案：A（0x08）

计算：pfn ^ (1 << 2) = 0x0C ^ 0x04

0x0C = 0000 1100

0x04 = 0000 0100

XOR = 0000 1000 = 0x08 = 8 ✅

原理：XOR 翻转第 order 位（第2位）。0x0C 的第2位是1，翻转为0，得到0x08。

物理含义：pfn=12 的 order=2 块占用页 12~15，其伙伴块占用页 8~11（pfn=8）。两块合并后形成 order=3 的块（页 8~15）。

Q20 综合诊断
下面是一段 "看起来正确" 的64位 clp2 实现。它在某些输入下会产生错误结果，请找出根本原因。
困难 
// 错误的64位 clp2 实现  
uint64_t clp2_buggy(uint64_t x) {  
    x--;  
    x |= x >> 1;  x |= x >> 2;  
    x |= x >> 4;  x |= x >> 8;  
    x |= x >> 16;              // ← 少了一步！  
    return x + 1;  
}
A没有问题，5步足以覆盖64位
B缺少 x |= x >> 32，对 x > 0xFFFFFFFF 的输入高32位无法传播，结果错误
Cx-- 对64位类型会产生未定义行为
Dreturn x+1 在64位下可能溢出
✅ 正确答案：B（缺少 x |= x >> 32）

32位 clp2 需要5步（覆盖32位 = 1+2+4+8+16）。

64位 clp2 需要6步（覆盖64位 = 1+2+4+8+16+32），必须追加 x |= x >> 32。

反例：x = 0x100000001（高32位有1-bit）

5步后高32位的1-bit只能传播到位32，低32位全1，但高32位仍不完整，返回错误结果。

规律：n 位 clp2 需要 log₂n 步 OR-Shift，32位5步，64位6步，128位7步。

正确的64位 clp2：在 x |= x >> 16 之后追加 x |= x >> 32;

🔥 第五组：最难挑战题 Q21–Q25
Q21 并行前缀原理
5步 OR-Shift 传播被称为"并行前缀"（Parallel Prefix）操作，由 Guy Steele 贡献。Warren 书中说"wherever you see parallel prefix, the material is due to Guy"。这一思想与哪个经典算法结构完全同构？
困难
A线性扫描（从左到右逐位传播）
B归并排序的分治结构（每步覆盖翻倍，O(log n) 步代替 O(n) 步）
C快速排序的随机分区
D动态规划的记忆化搜索
✅ 正确答案：B（归并排序分治结构）

5步 OR-Shift：每步覆盖范围翻倍（1→2→4→8→16→32位），O(log n) 步完成 n 位传播。

归并排序：每步合并范围翻倍（1→2→4→8→...→n），O(log n) 轮完成排序。

两者共享相同的"倍增分治"数学结构：每轮处理上一轮结果的两倍范围。

Warren 书致谢：Guy Steele Jr. 是 Java 和 Common Lisp 标准委员会核心成员，并行前缀思想被他系统性地引入位操纵领域。现代 GPU 并行 scan 算法也用同一思想。

Q22 C++20 标准
C++20 在 <bit> 头文件中新增了位操作标准函数，直接对应 Warren 书中 clp2/flp2 的功能。以下对应关系哪个是正确的？
进阶 
// C++20  新增函数
std::bit_floor(x);   // 对应 Warren 书中的 ???
std::bit_ceil(x);    // 对应 Warren 书中的 ???
std::bit_width(x);   // 对应 floor(log₂(x)) + 1
std::has_single_bit(x); // 对应 ???
Abit_floor = clp2，bit_ceil = flp2，has_single_bit = 判断2次幂
Bbit_floor = flp2，bit_ceil = clp2，has_single_bit = (x & (x-1)) == 0
Cbit_floor = flp2，bit_ceil = clp2，但 has_single_bit 没有对应的位操纵技巧
Dbit_floor 和 bit_ceil 只对2次幂有意义，其他输入是未定义行为
✅ 正确答案：B

std::bit_floor(x) = 不超过 x 的最大2次幂 = flp2(x)（floor to power of 2）

std::bit_ceil(x) = 不小于 x 的最小2次幂 = clp2(x)（ceiling to power of 2）

std::has_single_bit(x) = x 是2次幂且非零 = (x & (x-1)) == 0 && x != 0

现代编译器会把这三个函数编译为单条 CPU 指令：BSR（x86）、CLZ（ARM），性能远超5步 OR-Shift 版本。Warren 书中的位操纵算法已被 C++ 标准吸收。

Q23 系统设计
下面是一个真实面试题（来自某大厂系统组）：实现一个函数 is_aligned(ptr, align)，要求 align 必须是2次幂，若 align 不是2次幂则返回 -1（错误码）。哪个实现是完全正确的？
困难 
// 方案A
int is_aligned_A(void* ptr, size_t align) {
    if (align == 0 || (align & (align-1)) != 0)
        return -1;
    return ((uintptr_t)ptr & (align-1)) == 0;
}
// 方案B
int is_aligned_B(void* ptr, size_t align) {
    if ((align & (align-1)) != 0)
        return -1;
    return ((uintptr_t)ptr % align) == 0;
}
A方案A正确，方案B有缺陷：align=0 时 B 不报错（align-1 下溢，(0&0xFFF...)=0 通过了2次幂检测）且 %0 是未定义行为
B两者都正确，% 和 & 在 align 为2次幂时等价
C方案B正确，方案A的 align-1 对 size_t 类型会有符号扩展问题
D两者都错，应改用 __builtin_popcount(align) == 1
✅ 正确答案：A

方案B的两个缺陷：

① align=0 时：(0 & (0-1)) = 0 & 0xFFFF... = 0，条件 != 0 为 false，不报错，然后执行 ptr % 0，这是未定义行为（除零）！

② 即使 align 是2次幂，B 使用了 % 除法，性能较差。

方案A的设计：

① 先检查 align == 0（特判除零），再检查 (align & (align-1)) != 0（非2次幂），双重防御。

② 通过后用 & (align-1) 做对齐检测，纯位运算，高效。

这道题考察的正是 Warren 书 §3 的两个核心点：2次幂判断 + 对齐检测，以及 align=0 这个必须特判的边界。

Q24 综合推导
已知 x = 0xABCD1234，请计算 flp2(x) 和 clp2(x) 的值（32位无符号）。
最高有效位在哪里？0xABCD1234 的二进制最高位是第几位？
困难 
// 0xABCD1234 = 1010 1011 1100 1101 0001 0010 0011 0100
// 最高位是 bit 31（最左边的1在第31位）
// flp2(x) = ?   clp2(x) = ?
Aflp2 = 0x80000000（2³¹），clp2 = 0（溢出）
Bflp2 = 0x80000000（2³¹），clp2 = 0x100000000（2³²，超出32位表示范围）
Cflp2 = 0x40000000（2³⁰），clp2 = 0x80000000（2³¹）
Dflp2 = 0xABCD1234（本身是2次幂），clp2 = 0xABCD1234
✅ 正确答案：A

0xABCD1234 的最高有效位：0xA = 1010，最高位是 bit 31（从0计数）。

flp2：保留最高位，结果 = 2³¹ = 0x80000000 ✅

clp2：不小于 0xABCD1234 的最小2次幂 = 2³²。但 32位无符号无法表示 2³²，溢出回绕为 0x00000000 = 0。

B 是数学上正确但程序上错误的表述：2³² 无法存入 uint32_t，实际程序中 clp2 返回 0（这正是 Warren 书警告的溢出陷阱）。

判断：若 x 的最高位是 bit 31（即 x ≥ 2³¹），则 clp2(x) 在32位下必然溢出。使用前必须先检查 x <= 0x80000000U。

Q25 终极挑战
这是本章的"Boss题"。以下代码片段来自一个高性能内存分配器，综合使用了 Ch03 的多个技术。请完整分析它的功能，并指出注释中 ??? 处的正确值。
困难 
void* alloc(size_t req) {
    // Step 1: req=0 特判
    if (req == 0) req = 1;

    // Step 2: 向上对齐到 MIN_ALIGN=16
    req = (req + 15) & ~15UL;       // ← A: req=100→???

    // Step 3: 向上取2次幂（选择 size class）
    size_t cap = req - 1;
    cap |= cap >> 1;  cap |= cap >> 2;
    cap |= cap >> 4;  cap |= cap >> 8;
    cap |= cap >> 16; cap |= cap >> 32;
    cap = cap + 1;                    // ← B: cap=???

    // Step 4: 检测是否跨越 SLAB_SIZE=4096 边界
    uintptr_t base = get_slab_base();  // 假设=0xC000
    int cross = ((base ^ (base+cap-1)) >> 12) != 0;
                                       // ← C: cross=???
    if (cross) new_slab();
    return bump_ptr(cap);
}
// 输入 req=100, base=0xC000
// A=?, B=?, C=?
AA=112, B=128, C=0（不跨越）
BA=112, B=128, C=1（跨越）
CA=100, B=128, C=0（不跨越）
DA=112, B=256, C=0（不跨越）
✅ 正确答案：A（A=112, B=128, C=0）

Step A：(100 + 15) & ~15 = 115 & 0xFFFFFFF0 = 112 ✅

Step B：clp2(112)。112 = 2⁴ × 7，不是2次幂，不小于112的最小2次幂 = 128。

验证：111 → 5步OR-Shift → 0x7F = 127 → +1 = 128 ✅

Step C：base=0xC000，cap=128，末地址 = 0xC000 + 128 - 1 = 0xC07F。

0xC000 ^ 0xC07F = 0x007F，右移12位 = 0，不跨越 ✅

（0xC000 和 0xC07F 都在同一个4KB页 0xC000~0xCFFF 内）

这道题串联了 Ch03 的三个核心技术：向上对齐 → clp2 → 边界跨越检测，是真实分配器的微型复现。
