CHAPTER 03 · POWER-OF-2 BOUNDARIES · 2的幂边界

2 的幂边界

// 向下/向上取整到最近的2的幂 — 每个内存分配器的灵魂

这一章只做一件事：找到最近的2的幂。 听起来简单，但 Warren 用这件事串起了前导零计数（nlz）、位传播（smear）、 内存对齐、哈希表容量策略、SIMD 数据布局…… 每个操作系统内核、每个内存分配器、每个哈希表都在用这一章的技术， 只是大多数程序员不知道自己用的是什么。

flp2 / clp2 nlz 前导零 smear 传播 内存对齐 哈希表扩容 LZCNT / BSR buddy allocator
§3-1 定义与基本性质 — Definitions & Properties
// CORE DEFINITION flp2(x) = 不超过 x 的最大2的幂次 = 2^⌊log₂x⌋（x > 0），flp2(0) = 0。
clp2(x) = 不小于 x 的最小2的幂次 = 2^⌈log₂x⌉（x > 0），clp2(0) = 0。
注意：clp2(x) 对 x > 2³¹ 的结果在32位无符号中是 0（模2³²意义下）， Warren 明确要求"按模2³²的算术正确值"来定义。这个细节直接影响实现的正确性。
📋 flp2 / clp2 函数值对照表（Warren Table 3-1）
核心关系恒等式
// flp2 与 clp2 的相互转化关系 clp2(x) = flp2(x) // 当 x 是 2 的幂时 clp2(x) = flp2(x-1) * 2 // 当 x 不是 2 的幂 // 等价写法: clp2(x) = flp2(2x - 1) // x ≥ 1 flp2(x) = clp2(x) / 2 // 当 x 不是 2 的幂 // 与 nlz (前导零计数) 的关系: flp2(x) = 1 << (31 - nlz(x)) // x > 0 clp2(x) = 1 << (32 - nlz(x-1)) // x > 1 // 移位等价 (2^n 缩放): flp2(2ⁿ · x) = 2ⁿ · flp2(x) clp2(1/x) = 1 / flp2(x) // x ≠ 0
判断 2 的幂 — 5种方法对比
// ★ 方法1: Warren经典 (最快,2条指令) int is_pow2(unsigned x) { return x && !(x & (x - 1)); } // 方法2: popcount == 1 (需POPCNT指令) return __builtin_popcount(x) == 1; // 方法3: x == flp2(x) (需要先计算flp2) return x == flp2(x); // 方法4: clp2(x) == x return x == clp2(x); // 方法5: C++20 std::has_single_bit return std::has_single_bit(x); // 性能: 方法1 ≈ 方法2 >> 方法3/4 // 正确性: 所有方法对 x=0 需单独处理
§3-2 flp2 — 向下取整到最近2的幂
// THE SMEAR TECHNIQUE flp2 的实现原理极为精妙：先把 x 最高位的 1 向右"传播"（smear）填满所有低位， 得到一个形如 0111...1 的数，然后再右移1位并加1……不对， Warren 的方法是：把最高位1传播到所有低位后，再做 result - (result >> 1)。 但更简洁的方式是：smear 之后再右移1位再加1， 或者直接 (smear >> 1) + 1。 这个"smear-then-halve"的技巧把对数运算变成了纯位操作，理解它需要看透补码的传播本质。

// flp2: 向下取整到最近2的幂 (Warren §3-2) // 方法: OR-smear + 右移 (5次移位OR, 1次右移减法) unsigned flp2(unsigned x) { x = x | (x >> 1); // 传播最高位往右1位 x = x | (x >> 2); // 再传播2位 x = x | (x >> 4); // 再传播4位 x = x | (x >> 8); // 再传播8位 x = x | (x >> 16); // 再传播16位 → 最高位1已蔓延到所有低位 // 此时 x = 0...01111...1 (最高位1右边全是1) return x - (x >> 1); // 去掉低位的1，只留最高位 ✓ } // 总计: 10条指令 (5×OR, 5×SHR, 1×SUB, 1×SHR) // x=0时返回0 ✓ (无需特判) // 64位版本 (再加两步即可): unsigned long long flp2_64(unsigned long long x) { x |= (x >> 1); x |= (x >> 2); x |= (x >> 4); x |= (x >> 8); x |= (x >> 16); x |= (x >> 32); return x - (x >> 1); }
// 方法2: 利用 nlz (前导零计数) — 只需2条指令! unsigned flp2_fast(unsigned x) { if (x == 0) return 0; return 0x80000000u >> __builtin_clz(x); // __builtin_clz(x) = nlz(x) = 前导零个数 // 0x80000000 >> nlz(x) = 2^(31-nlz(x)) = flp2(x) } // 方法3: 用 BSR 指令 (x86 Bit Scan Reverse) // BSR eax, x → eax = 31 - nlz(x) = floor(log2(x)) unsigned flp2_bsr(unsigned x) { if (x == 0) return 0; int pos; __asm__ ("bsrl %1, %0" : "=r"(pos) : "r"(x)); return 1u << pos; } // C++20 最简洁写法: return std::bit_floor(x); // <bit> header // 性能对比 (cycles, Skylake): // smear方法: ~5 cycles (no LZCNT) // nlz方法: ~3 cycles (with LZCNT) // bit_floor: ~2 cycles (编译器优化后)
; x86-64: GCC -O2 flp2(x) → (有LZCNT时) lzcnt ecx, edi ; ecx = 前导零数 mov eax, 1 shl eax, cl ; eax = 2^(31-nlz) = flp2... 不对 ; 实际: 0x80000000u >> clz(x) lzcnt ecx, edi mov eax, 0x80000000 shr eax, cl ; ✓ 2条指令 ; x86-64: 无LZCNT时 (旧CPU) 用BSR bsr ecx, edi ; ecx = 31 - nlz(x) mov eax, 1 shl eax, cl ; ✓ 3条指令 ; ARM64: CLZ + LSL clz w1, w0 ; w1 = nlz(x) mov w2, #0x80000000 lsr w0, w2, w1 ; ✓ 3条指令 ; RISC-V: 需要软件smear或扩展(Zbb: CLZ) clz a1, a0 ; (Zbb扩展) li a2, 0x80000000 srl a0, a2, a1 ; ✓
§3-3 clp2 — 向上取整到最近2的幂
// THE CLASSIC TRICK clp2 是 flp2 的"镜像"：先把 x-1 的最高位1向右传播到所有低位得到全1掩码， 然后加1进位得到下一个2的幂。这是哈希表、内存分配器、SIMD 缓冲区分配中 出现频率最高的操作之一。Linux 内核的 roundup_pow_of_two()、 jemalloc 的 size class 计算、Go runtime 的 growslice 都用了这个技术。

// clp2: 向上取整到最近2的幂 (Warren §3-3) // 思路: (x-1) smear填满低位 → 加1 → 进位得到2的幂 unsigned clp2(unsigned x) { x = x - 1; // 关键: 先减1，避免x本身是2的幂时多进一位 x = x | (x >> 1); x = x | (x >> 2); x = x | (x >> 4); x = x | (x >> 8); x = x | (x >> 16); // smear → 0...0111...1 return x + 1; // +1 → 进位得到下一个2的幂 } // 总计: 12条指令 (1×SUB, 5×OR, 5×SHR, 1×ADD) // 特殊值: clp2(0)=0 ✓ clp2(2³¹+1)=0 (模2³²溢出) // ⚠ 对 x > 2³¹ 结果溢出为 0 (Warren 明确定义如此) // Linux kernel 等价实现: static inline unsigned long roundup_pow_of_two(unsigned long n){ return 1UL << fls_long(n - 1); // fls_long = "find last set" = 31 - nlz }
// 快速版本: 用 nlz 只需 3 条指令 unsigned clp2_fast(unsigned x) { if (x <= 1) return 1; return 1u << (32 - __builtin_clz(x - 1)); // 32 - nlz(x-1) = ⌈log₂x⌉ } // C++20 std::bit_ceil: return std::bit_ceil(x); // <bit> 头文件 // Python 等价 (位操作版): // def clp2(x): return 1 if x==0 else 1 << (x-1).bit_length() // Go 等价 (bits.Len): // func clp2(x uint) uint { return 1 << bits.Len(x-1) } // 性能 (Skylake, 含特判): // smear方法: ~6 cycles // nlz方法: ~4 cycles (LZCNT available) // bit_ceil: ~2 cycles (编译器最优化)
// ★ 应用1: 哈希表容量 (HashMap 初始化) // 保证容量总是2的幂，使得 h % cap == h & (cap-1) int cap = clp2(requested_size * 4 / 3); // 75%负载因子 int idx = hash(key) & (cap - 1); // 比 % 快很多 // ★ 应用2: Buddy Allocator (Linux 内存系统) int order = ilog2(clp2(size)); // 找到需要的阶数 page_block = alloc_pages(GFP_KERNEL, order); // ★ 应用3: SIMD 缓冲区对齐分配 // AVX-512需要64字节对齐，确保大小也是64的倍数 size_t aligned_size = clp2(size); void* buf = aligned_alloc(64, aligned_size); // ★ 应用4: Go runtime slice 扩容 (growslice) // 当 cap < 256 时，直接 clp2(cap) // 这是Go slice append性能的核心 // ★ 应用5: GPU纹理尺寸 (OpenGL/Vulkan) // 传统GPU要求纹理宽高必须是2的幂 (MipMap需要) int tex_w = clp2(img_w), tex_h = clp2(img_h);

§3-4 对齐操作 — Alignment to Powers of 2
// ALIGNMENT IS EVERYWHERE 内存对齐到2的幂次是系统编程中最高频的操作之一： struct 对齐、DMA 缓冲区、SIMD 数组、缓存行、页表项……全部都要对齐到某个2的幂。 核心公式只有两个：向下对齐 = x & ~(a-1)， 向上对齐 = (x + a-1) & ~(a-1)。 理解这两个公式，就理解了 C 编译器里 99% 的对齐代码。
向下对齐 (Floor Align)
// 向下对齐到 a 的倍数 (a必须是2的幂) // align_down(x, a) = a * ⌊x/a⌋ unsigned align_down(unsigned x, unsigned a) { return x & ~(a - 1); // 清除低 log₂(a) 位 } // 等价: x - (x % a) 但上面更快 // 示例: align_down(0x1007, 16) // = 0x1000 align_down(0x1000, 16) // = 0x1000 (已对齐) align_down(0x1FFF, 4096) // = 0x1000 (页对齐) // C11 宏写法 (GCC扩展 __typeof__): #define ALIGN_DOWN(x, a) \ ((x) & ~((__typeof__(x))(a) - 1))
向上对齐 (Ceiling Align)
// 向上对齐到 a 的倍数 (a必须是2的幂) // align_up(x, a) = a * ⌈x/a⌉ unsigned align_up(unsigned x, unsigned a) { return (x + a - 1) & ~(a - 1); // = (x + a-1) 清除低位 } // 等价: (x + a-1) / a * a (但有溢出风险) // 示例: align_up(0x1007, 16) // = 0x1010 align_up(0x1000, 16) // = 0x1000 (已对齐) align_up(5, 8) // = 8 // Linux 内核标准宏: #define ALIGN(x, a) \ (((x) + (a) - 1) & ~((a) - 1)) // arch/x86/include/asm/page.h 等处使用

实战场景：各对齐粒度的工业用途
align = 4 (32-bit word)
C struct 默认对齐。ARM Cortex-M 硬件要求 32 位访问必须 4 字节对齐，否则 HardFault。malloc() 返回的指针至少 4 字节对齐。
align = 16 (SSE/NEON)
SSE2 的 _mm_load_si128 要求 16 字节对齐（_mm_loadu 不要求但慢 ~2x）。posix_memalign(p,16,n) 保证此对齐。
align = 64 (AVX-512/cacheline)
Intel x86 cacheline = 64 字节。AVX-512 zmm 寄存器 512-bit 需要 64 字节对齐。jemalloc/tcmalloc 小内存块以 64 字节对齐分配。
align = 4096 (page)
mmap() 返回页对齐地址。DMA 缓冲区必须页对齐（O_DIRECT I/O）。Huge Page 需要 2MB/1GB 对齐。Linux vm_area_struct 基于页粒度管理。
§3-5 nlz — 前导零计数与 ilog2
// WHY nlz IS THE KEY PRIMITIVE nlz(x) = 前导零的个数（Number of Leading Zeros），等价于 31 - ⌊log₂x⌋。 它是 flp2、clp2、整数对数、比较谓词的统一入口。 Warren 用整整一节来讨论它，因为在没有 CLZ/LZCNT 硬件指令的年代（2002年前的大多数 RISC）， 用软件实现 nlz 是整个第3章的核心挑战。 理解这些软件实现方式，就理解了为什么 BMI1 指令集在2013年出现后会让如此多的代码变短。

二分搜索法求 nlz 的每步判断:
// 方法1: 二分搜索法 (Warren §5-3, 用于无CLZ指令的平台) // 总计约 ~20 条基础RISC指令，无除法无循环 int nlz(unsigned x) { int n = 0; if (x == 0) return 32; if (x <= 0x0000FFFF) { n += 16; x <<= 16; } // 高16位全0? if (x <= 0x00FFFFFF) { n += 8; x <<= 8; } // 高8位全0? if (x <= 0x0FFFFFFF) { n += 4; x <<= 4; } // 高4位全0? if (x <= 0x3FFFFFFF) { n += 2; x <<= 2; } // 高2位全0? if (x <= 0x7FFFFFFF) { n += 1; } // 高1位全0? return n; } // 注: 也可完全无分支 (用减法+算术右移掩码代替if) // 分支版本在预测准确时更快; 无分支版本在SIMD中更好
// 方法2: 利用 IEEE 754 浮点格式提取指数 (GCC扩展技巧) // 原理: float(x) 的指数字段直接就是 ⌊log₂x⌋ + 127 // ⚠️ 严格来说是 UB (type punning), 但几乎所有编译器都支持 int nlz_float(unsigned x) { union { unsigned u; float f; } t; t.f = (float)x; return 158 - (t.u >> 23); // 158 = 127(偏置) + 31(位宽-1) // t.u >> 23: 提取8位指数字段 } // x=0 时返回 159-127=32 ✓ (浮点0的指数=0) // 速度: 依赖整数↔浮点路径延迟 (~4-6 cycles on Skylake) // 精度: 当 x 超过 float 尾数精度(2^24)时可能不准！ // → Warren 明确警告: 仅对 x < 2^24 可靠 // 安全的 C99 版本 (memcpy 不是UB): int nlz_safe(unsigned x) { float fx = (float)x; unsigned u; memcpy(&u, &fx, sizeof(u)); return 158 - (u >> 23); }
// 方法3: 256项查表法 (用空间换时间) // 把32位分成4个字节，每字节查表得到前导零数 static const char nlz8[256] = { 8,7,6,6,5,5,5,5, 4,4,4,4,4,4,4,4, 3,3,3,3,3,3,3,3, 3,3,3,3,3,3,3,3, 2,2,2,2,2,2,2,2, 2,2,2,2,2,2,2,2, 2,2,2,2,2,2,2,2, 2,2,2,2,2,2,2,2, /* ... 剩余192项均为1或0 ... */ }; int nlz_table(unsigned x) { if (x >> 24) return nlz8[x >> 24]; if (x >> 16) return nlz8[x >> 16] + 8; if (x >> 8) return nlz8[x >> 8] + 16; return nlz8[x] + 24; } // 优点: 4次比较+1次查表+加法，约5条指令 // 缺点: 256字节查表污染L1 cache (特别是在热循环中) // 适合: 代码密度敏感、L1 cache 压力小的场景
// 方法4: 直接使用硬件指令 (现代平台最优解) // GCC/Clang 内建 (最推荐, 编译器选最优指令): int nlz = __builtin_clz(x); // x ≠ 0 // x=0 时行为未定义! 需手动处理 int nlz_safe = x ? __builtin_clz(x) : 32; // C++20 标准库 <bit> (最可移植): int nlz = std::countl_zero(x); // x=0 时返回32 ✓ // x86-64 汇编 (两种指令): // BSR (Bit Scan Reverse): 结果 = 31-nlz, x=0时ZF=1结果不定 bsr eax, edi ; eax = 31 - nlz(x) xor eax, 31 ; eax = nlz(x) // LZCNT (BMI1, 2013+): x=0时返回32 ✓ 更安全 lzcnt eax, edi ; eax = nlz(x), 1 cycle latency // ARM64: CLZ (单指令, 1-3 cycles) clz w0, w0 ; ✓ x=0时返回32 // RISC-V Zbb 扩展: clz a0, a0 ; Zbb 必需扩展 // 历史: 在2013年BMI1之前, x86唯一的近似是BSR (不安全) // 这就是为什么Warren要花整节讲软件实现

§HIST 历史演变 — 为什么这些技术以这种方式出现
1960年代 — IBM System/360
硬件对齐强制约束催生软件技巧
S/360 的内存访问要求 halfword 在2的倍数地址、word 在4的倍数地址， 否则触发 Alignment Exception（0x06 中断）。 这是"向上对齐"公式 (x+a-1)&~(a-1) 进入工业代码库的直接原因——程序员必须手动计算对齐偏移。 Warren 在 IBM 工作了数十年，这个背景是他对对齐如此重视的根源。
1986 — RISC 浪潮 & Power 架构
没有 CLZ 指令，软件 nlz 成为必需
RISC 设计哲学追求指令集精简，CLZ（Count Leading Zeros）被认为是"不够通用"的操作， 大多数早期 RISC ISA（MIPS R2000、SPARC V7）都没有它。 IBM POWER 架构的 cntlz 指令是少数例外之一（1990年）。 Warren 书中的二分搜索法 nlz 正是为这个时代写的——在当时是真实的工程需求，不是学术玩具。
1999 — MIPS32 / ARMv5
CLZ 指令普及到主流 RISC
MIPS32/64 ISA 加入 CLZ/CLO 指令（Count Leading Zeros/Ones）， ARMv5T 加入 CLZ 指令（用于支持 TCP 校验和加速）。 至此，主流嵌入式和服务器 RISC 都有了硬件 nlz。 但 x86 在 2002 年（Pentium 4 时代）才通过 BSR 近似支持， 真正的 LZCNT 要等到 2013 年 Haswell 的 BMI1。
2002 — 《Hacker's Delight》第1版
Warren 把"最近的2的幂"系统化
第1版出版时，Linux 内核的 roundup_pow_of_two() 已经存在但实现各有差异。Warren 第3章把 flp2/clp2 作为正式的数学函数定义， 并给出了多种实现的权衡分析。这种系统化的处理方式深刻影响了后来 GCC 4.x 中 __builtin_clz 的推广和使用模式。
2013 — Intel BMI1 + C++20 <bit>
软件技巧成为标准库，硬件支持完备
BMI1 的 LZCNT 指令让 nlz 在 x86 上也变成单指令。 C++20 标准库 <bit> 头文件引入 std::bit_floor（即 flp2）、std::bit_ceil（即 clp2）、 std::countl_zero（即 nlz）——把第3章所有核心函数变成标准库的一部分。 Warren 的第3章可以视为 C++ 标准委员会在2020年"认可"的历史遗产。

第3章的本质：把"找最近的2的幂"变成一种思维习惯

为什么选择"smear后减半"而不是直接用 log₂？

这个问题触及了整本书的核心哲学。在整数计算中，log₂ 是浮点操作，有精度损失（Warren §5-3的浮点法就说明了这一点，大数不准）。 Smear 方法完全在整数域内，没有舍入，没有浮点单元依赖，在任何平台都能正确运行。 这是一种系统程序员的本能：优先寻找整数域内的精确算法，而不是借用浮点的"近似"。 那些习惯了高层语言的程序员很难理解这种本能的价值，直到某天他们调试一个 因为浮点精度导致内存越界的 bug，才会明白为什么 Warren 要把 12 条整数指令的方案 摆在 3 条浮点指令的方案之前。

对齐公式的陷阱：为什么不用 % 运算符？

(x + a - 1) / a * a 是数学上正确的向上对齐， 但它有两个问题：1. 对于非2的幂的 a，这个表达式涉及整数除法（慢，20-80 cycles）； 2. x + a - 1 可能溢出（当 x 接近 UINT_MAX 时）。 而 (x + a - 1) & ~(a - 1) 把除法变成位运算（1 cycle）， 且限制 a 必须是2的幂这个约束恰好保证了位运算的正确性。 这是一个"以约束换速度"的典型案例——工程上你强制要求对齐粒度是2的幂，就能获得10倍的速度。 Linux 内核、jemalloc、tcmalloc 都遵守这个约定。

clp2 的溢出设计：不是bug，是有意为之

Warren 明确定义 clp2(x) = 0 当 x > 2³¹（32位模运算意义下）。 很多人第一次看到这个会觉得"这不是bug吗"。但这个设计是合理的工程决策： 在32位系统中，能分配超过2GB内存的场景本来就不存在； 而在64位系统中你应该用 uint64_t 版本。 强行返回"正确值"（2³²）会要求用更大的返回类型，破坏接口一致性。 Rust 的 checked_next_power_of_two() 用 Option<u32> 处理这个边界——这是更现代的做法，但代价是调用者必须处理 None。 不同的工程哲学，各有利弊，没有绝对的对错。

这些技巧在 GPU/并行计算时代还有价值吗？

有，而且更有。CUDA/HIP 的线程块大小（blockDim）必须是2的幂， 共享内存的 bank conflict 分析依赖2的幂对齐，warp 大小（32）本身就是2的幂。 Metal/Vulkan 的 buffer 对齐要求同样基于2的幂。 nlz 在 GPU 上对应 __clz()（CUDA 内建函数）， 在 warp 级别的 ballot 操作中计算最高有效位位置时也会用到。 第3章的所有技巧在 GPU 编程中的使用频率，不亚于在 CPU 系统编程中的使用频率。

可以推广到哪里？量子计算的视角

在量子计算中，qubit 数量（即量子寄存器宽度）通常也取2的幂次， 量子 Fourier 变换（QFT）的电路深度是 O(n²)，其中 n = log₂(N)，N 是2的幂次。 更有意思的是，量子纠错码（如 surface code）的逻辑量子比特数量也往往是2的幂次对齐的， 因为这与解码电路的并行性直接相关。 从 IBM S/360 的硬件对齐约束，到量子处理器的电路拓扑约束， 2的幂的偏好是计算机科学中最深层的结构性偏好之一， 根植于二进制表示和"分而治之"的算法范式。 Warren 第3章的技术细节会过时，但它背后的数学结构会一直在。
