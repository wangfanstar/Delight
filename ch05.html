位计数

// popcount · parity · nlz · ntz · divide & conquer in silicon

第5章是 Hacker's Delight 中最具传奇色彩的一章。 "如何数一个整数里有多少个1" 这个问题， 从 1960 年 IBM Stretch 的硬件指令，到 HAKMEM 备忘录里的黑客游戏， 到 x86 的 POPCNT 指令，到 RISC-V 的 B 扩展—— 跨越六十年的工程史，每一代程序员都在用更少的指令解决同一个问题。 这一章还包含前导零（nlz/CLZ）、尾随零（ntz/CTZ）和奇偶校验（parity）—— 四个看似简单却深藏精妙的位操作原语，是密码学、压缩、网络协议和 CPU 设计的共同基石。

§5-1 计数1-位 — Counting 1-Bits (Population Count)
// THE CLASSIC: 分治法逐步归并相邻位的和 Warren 介绍的核心方法是分治策略（Divide & Conquer）： 先把32位字拆成16个2位字段，每个字段存储对应2位中1的个数； 再合并相邻2位字段为4位字段，依此类推，共 log₂(32) = 5 步完成。 这不是一个人想出来的技巧——它同时出现在 1960 年代的硬件电路设计和 HAKMEM memo（1972）里， 是并行计算思维在单个寄存器上的缩影：把串行问题转化为树形并行归并。

📚 popcount 的四种经典实现——从朴素到最优
// 方法1: 朴素循环 — O(32) 最慢, 但最直观 int pop_naive(unsigned x) { int n = 0; while (x) { n += x & 1; // 检查最低位 x >>= 1; // 右移 } return n; } // 指令数: ~64 条 (最坏情况) // 缺点: 分支 + 循环, 无法流水线化 // 变体: Kernighan 技巧 (只循环1-bit个数次) int pop_kernighan(unsigned x) { int n = 0; while (x) { x &= x - 1; // 清除最低1位 n++; } return n; } // 比朴素快 (稀疏1-bit时), 但仍有分支 // 历史: 出自 Kernighan & Ritchie《C程序设计语言》练习
// 方法2: Warren §5-1 分治法 — 21条指令, 无分支 // Figure 5-2: 最优无硬件支持版本 int pop(unsigned x) { // 步骤1: 将x替换为"2位字段中1的个数" // x -= (x>>1) & 0x55555555 // 等价于: 每对 [a,b] → [0,a+b] (a,b∈{0,1}) x = x - ((x >> 1) & 0x55555555); // 步骤2: 合并相邻2位字段 → 4位字段 x = (x & 0x33333333) + ((x >> 2) & 0x33333333); // 步骤3: 合并4位字段 → 8位字段, 并消除高位 x = (x + (x >> 4)) & 0x0F0F0F0F; // 步骤4: 合并字节 (字节内最大值8, 不会进位到高字节) x = x + (x >> 8); // 字节1+字节0 → 高16位字节 x = x + (x >> 16); // 合并两个16位半字 return x & 0x0000003F; // 只取低6位 (0~32) } // 魔法常数的含义: // 0x55555555 = 0101 0101 ... 0101 (选奇数位) // 0x33333333 = 0011 0011 ... 0011 (选每4位中低2位) // 0x0F0F0F0F = 0000 1111 ... 1111 (选每字节低4位) // 0x0000003F = 0...0 0011 1111 (最终掩码) // 步骤1的妙处: x - ((x>>1) & 0x55555555) // 原始码: ab (a,b ∈ {0,1}) // (x>>1) & 0x55: 取a (高位)到低位 // x - (x>>1)&0x55 = 2a+b - a = a+b ✓ // 比 (x&0x55) + ((x>>1)&0x55) 少一条指令
// 方法3: HAKMEM memo (1972) item 169 — mod 63 算法 // 出自 MIT AI Lab, Gosper/Beeler/Schroeppel int pop_hakmem(unsigned x) { unsigned n; n = (x >> 1) & 033333333333; // 八进制! 0x1B6DB6DB x = x - n; n = (n >> 1) & 033333333333; x = x - n; // 现在x是 3-bit 字段的bitcount之和 x = (x + (x >> 3)) & 030707070707; // 6-bit字段 return x % 63; // 6位字段之和 ≡ 总和 (mod 63) } // 原理: 6-bit最大值63 = 2^6-1 = 0x3F // 一个32位整数 = 5个6位字段 + 2位余量 // 各6位字段之和 mod 63 = 全部之和 // 因为 64 ≡ 1 (mod 63) → 位权在mod63下等价 // 缺点: 模运算在无硬件除法的机器上昂贵 // 优点: 在有快速mod的机器上代码极短 // 64-bit 版: 使用 mod 255 // 类似地: 256 ≡ 1 (mod 255) // 8个8-bit字段之和 mod 255 = 总1-bit数 int pop64_mod255(unsigned long long x) { x -= (x >> 1) & 0x5555555555555555ULL; x = (x & 0x3333333333333333ULL) + ((x >> 2) & 0x3333333333333333ULL); x = (x + (x >> 4)) & 0x0F0F0F0F0F0F0F0FULL; return (x * 0x0101010101010101ULL) >> 56; // 乘以全1字节向量 → 将8个字节累加到最高字节 }
// 方法4: 硬件指令 — 最快, 1-3 周期 // C/C++: GCC/Clang 内建 int pop_hw(unsigned x) { return __builtin_popcount(x); // GCC 会根据目标架构选择: // x86 with SSE4.2: POPCNT eax, edi (1指令) // ARM v8-A: CNT + ADDV (2指令) // RISC-V B-ext: CPOP rd, rs1 (1指令) } // x86 POPCNT (Intel Nehalem 2008 起): // POPCNT r64, r/m64 — 1 cycle latency, 1 cycle throughput // 编译: gcc -mpopcnt -O2 // 注意: 有"虚假依赖"bug (Haswell前), 目标寄存器需清零 // ARM64 (AArch64): // CNT v0.8b, v0.8b — 字节级popcount (SIMD) // ADDV b0, v0.8b — 水平加法归并字节 // 总计2条SIMD指令, 可批量处理 // Java: // Integer.bitCount(x) // JIT自动映射到POPCNT // Long.bitCount(x) // 64位版本 // Rust: // x.count_ones() // 标准库, 自动POPCNT // Python: // bin(x).count('1') // 朴素, 但CPython优化了字符串 // int.bit_count() // Python 3.10+ 原生方法 // 性能对比 (Skylake, 32-bit): // POPCNT: ~1 cycle // Warren §5-1: ~8 cycles (21 instructions) // HAKMEM: ~12 cycles (含mod) // 朴素循环: ~50 cycles (平均16个1)
⚡ 进阶：进位保存加法器（CSA）批量 popcount — 大数组的最优解
// CARRY-SAVE ADDER — 硬件电路思维迁移到软件 当需要对一个大数组求总 popcount 时，Warren §5-1 给出了基于 进位保存加法器（CSA, 3:2 压缩器）的方案。 CSA 将三个输入压缩为两个（sum + carry），在软件中用 5 条逻辑指令模拟一个完整加法器。 每次迭代处理8个数组元素，相比朴素方法（每个元素单独调用 pop()）减少了约 40% 的指令数。 这个方法的起源是硬件乘法器电路——Harley 和 Seal 在1996-97年将其迁移到软件 popcount。
// §5-1 CSA (3:2 压缩器) 宏: // 将三个32位字 a,b,c 压缩为 (h=进位, l=和) // 全加器逻辑 (每位独立并行): // h = (a&b) | (a&c) | (b&c) = ab|(a^b)c // l = a ^ b ^ c // CSA 用5条逻辑指令在32个独立1-bit全加器上并行运行 // 对大数组 A[n] 求总popcount, 8元素一组: long long array_pop8(unsigned *A, int n) { long long total = 0; unsigned ones = 0, twos = 0, fours = 0, eights = 0; int i; for (i = 0; i <= n - 8; i += 8) { unsigned t1, t2, t3, t4; // CSA1: ones, A[i], A[i+1] → twos_ab, ones t1 = ones ^ A[i]; t2 = A[i+1]; unsigned twos_ab = (ones & A[i]) | ((t1) & t2); ones = t1 ^ t2; // CSA2: twos_ab, A[i+2], A[i+3] → fours_ab, twos_cd t1 = twos_ab ^ A[i+2]; t3 = A[i+3]; unsigned fours_ab = (twos_ab & A[i+2]) | (t1 & t3); unsigned twos_cd = t1 ^ t3; // ...后续 CSA3-7 类似 (处理 A[i+4..7]) // 最终: total += 8*pop(eights)+4*pop(fours)+2*pop(twos)+pop(ones) total += pop(fours_ab); // 简化示意 } // 处理剩余元素 for (; i < n; i++) total += pop(A[i]); return total + pop(ones) + 2*pop(twos) + 4*pop(fours); } // 性能: ~12.33 指令/字 vs 朴素 ~16 指令/字 // 现代: AVX-512 VPOPCNTDQ 每周期处理 16×32bit = 512bit!

§5-2 奇偶校验 — Parity
// PARITY = popcount(x) & 1 = XOR of all bits 奇偶校验（parity）就是"1-bit的个数是奇数还是偶数"，等价于所有位的 XOR。 这个等价性是并行前缀（Parallel Prefix）计算的完美例子： XOR 是结合律运算，32位的 XOR 归约可以用 5 步完成（每步折叠一半）。 Warren 指出这与 Gray 码的性质深度关联： Gray(x) 的第 i 位 = parity(x ≫ i)， 格雷码转二进制的逆运算本质上就是前缀 XOR。
奇偶校验实现（并行前缀 XOR）
// §5-2 奇偶校验: 返回 popcount(x) % 2 // 方法1: 直接调用 popcount int parity_pop(unsigned x) { return pop(x) & 1; } // 方法2: 并行前缀 XOR (Warren §5-2, 5步) int parity(unsigned x) { x ^= x >> 16; // 折叠高16位到低16位 x ^= x >> 8; // 折叠高8位到低8位 x ^= x >> 4; // 折叠高4位到低4位 x ^= x >> 2; x ^= x >> 1; return x & 1; // 取最低位 } // 共 5×2 = 10 条指令, 无分支, 可流水线 // 方法3: 查表 (适合8-bit块) // 预计算 parity_table[256] int parity_table(unsigned x) { x ^= x >> 16; x ^= x >> 8; return parity_table[x & 0xFF]; // 只需2次折叠+1次查表 } // 方法4: 硬件 // x86: POPCNT + AND 1 (无直接parity指令) // x86 EFLAGS: PF (Parity Flag) 给出最低字节奇偶性 // 需要折叠后再用 LAHF/SETPE 取PF位 // GCC: __builtin_parity(x) — 最优选择

📦 奇偶校验的实际应用场景
💾 ECC 内存
DDR 内存的 ECC（Error-Correcting Code）使用汉明码，其核心是对数据位的多组奇偶校验。每64字节数据附加8字节 ECC，能检测2位错误并纠正1位错误。服务器内存每次写操作都要计算这些校验位。
📡 串行通信 (UART/RS-232)
UART 协议可选奇偶校验位（Even/Odd/None），每帧数据在停止位前附加1位奇偶校验。硬件 UART 控制器用 XOR 树实现，完全并行计算，不影响波特率。
🔐 密码学 (GF(2) 运算)
GF(2) 上的多项式运算本质是 XOR。AES 的 GF(2⁸) 乘法、CRC 校验、LDPC 纠错码都大量使用奇偶校验的变体。现代 CPU 专为此提供 PCLMULQDQ 指令（进位less乘法 = XOR 多项式乘）。
🧬 Gray 码转换
Gray 码转二进制 = 并行前缀 XOR（§13-1）。Warren 指出这与奇偶校验的计算结构完全相同，只是把"折叠后取最低位"换成了"保留每步的结果"。前缀 XOR 是可逆的，逆运算仍是前缀 XOR。
§5-3 前导零计数 — Counting Leading Zeros (nlz / CLZ)
// nlz(x) = 最高1位之前有多少个0 nlz(x)（number of leading zeros）告诉你最高有效位在哪里， 等价于 31 - floor(log₂(x))（x>0时）。 这个函数在整数对数计算、规范化、IEEE浮点指数提取、哈希表大小舍入等场景无处不在。 Warren 给出了三条路径：二分搜索法（5步，21条指令）、 浮点指数借用法（把整数转为 double，提取指数字段）、 以及De Bruijn 乘法查表法。每种方法都代表一种完全不同的计算哲学。

// §5-3 二分搜索法 nlz — Warren Figure 5-12 // 5步确定最高1位, 共21条指令, 无分支 int nlz(unsigned x) { int n; if (x == 0) return 32; n = 0; // 每步缩小搜索范围一半 if (x <= 0x0000FFFF) { n += 16; x <<= 16; } if (x <= 0x00FFFFFF) { n += 8; x <<= 8; } if (x <= 0x0FFFFFFF) { n += 4; x <<= 4; } if (x <= 0x3FFFFFFF) { n += 2; x <<= 2; } if (x <= 0x7FFFFFFF) { n += 1; } return n; } // 分支版本: 最坏5次分支 // 无分支版本 (用掩码代替if): int nlz_branchless(unsigned x) { int n = 32; unsigned y; y = x >>16; if(y) { n-=16; x=y; } y = x >> 8; if(y) { n-= 8; x=y; } y = x >> 4; if(y) { n-= 4; x=y; } y = x >> 2; if(y) { n-= 2; x=y; } y = x >> 1; if(y) { n-= 1; x=y; } return n - x; }
// §5-3 浮点指数借用法 — Warren Figure 5-19 // 把整数转为 IEEE 754 double, 提取指数字段 // IEEE double: [符号1位][指数11位][尾数52位] // 如果 x > 0: double(x) 的指数 = 1023 + floor(log2 x) // 因此: nlz(x) = 31 - floor(log2 x) = 1054 - exponent // 注意: 以下代码依赖 type-punning, 严格说是 UB // C++20 可用 std::bit_cast 替代 int nlz_float(unsigned k) { union { unsigned asInt[2]; double asDouble; }; int n; asDouble = (double)k + 0.5; // +0.5 处理k=0 // LE=1(小端): 指数在 asInt[1] 的高11位 n = 1054 - (asInt[1] >> 20); return n; } // 原理: 1023 + floor(log2 k) = 指数字段 // 1054 - 指数 = 31 - floor(log2 k) = nlz(k) ✓ // 32位float版 (指数8位, 偏移127): // asFloat = (float)k + 0.5f // n = 158 - (asInt >> 23) // 158 = 127+31 // 缺点: 整数→浮点转换慢(5-10周期) // 跨寄存器组的数据移动代价高 // 现代 CPU 用 LZCNT 只需 1 周期
// §5-3 De Bruijn 乘法查表法 — Harley's Algorithm // Warren Figure 5-17, 改进版 Figure 5-18 // 原理: De Bruijn 序列 — 每个长度k的子串恰好出现一次 // 将 nlz 转化为查表: 先用乘法提取"指纹", 再查6位表 // 步骤1: 将 x 变为只有最高1位的形式 (类似 flp2) // 步骤2: 乘以 De Bruijn 常数, 取高6位作为表索引 // 步骤3: 查 64 项表, 返回 nlz 值 static const char table[64] = { 32,20,19, u, u,18, u, 7, 10,17, u, u,14, u, 6, u, u, 9, u,16, u, u, 1,26, u,13, u, u,24, 5, u, u, u,21, u, 8,11, u,15, u, u, u, u, 2,27, 0,25, u, 22, u,12, u, u, 3,28, u, 23, u, 4,29, u, u,30,31 }; int nlz_debruijn(unsigned x) { x = x | (x >> 1); // 将最高1位以下全部置1 x = x | (x >> 2); x = x | (x >> 4); x = x | (x >> 8); x = x | (x >> 16); // x 现在是 0...01...1 形式 x = ~x; // 取反: 1...10...0 x = x - (x >> 1); // 只保留最高1位 return table[(x * 0x045BCED1) >> 26]; // 0x045BCED1 是 De Bruijn 常数 // = 17 × 65 × 129 × 513 (mod 2^32) // 乘法将最高1位的位置映射到高6位 // 高6位恰好在0~63内唯一, 因此可以查表 } // 指令数约15, 含1次乘法+1次内存访问 // 若无乘法: 改用 Goryavsky 变体, 纯移位加法
// §5-3 硬件 nlz 指令 — 最优解 // x86 — BSR (Bit Scan Reverse) + LZCNT // BSR dst, src: dst = 最高1位的位置 (0-indexed from LSB) // LZCNT dst, src: dst = 前导零个数 (BMI1, Intel Haswell+) // 注意: BSR在x=0时行为未定义! 📦 nlz 的核心应用场景
📐 整数 floor(log₂ x) = 31 - nlz(x)
哈希表容量舍入到2的幂、B树阶数计算、IEEE浮点规范化移位量、Huffman树深度计算，均依赖这一恒等式。
🔢 向上取整到2的幂 = 1 << (32 - nlz(x-1))
malloc对齐、GPU着色器缓冲区大小、网络数据包分段大小，都需要快速计算"不小于x的最小2的幂"。Warren §3-2 的 clp2 就是此公式。
🌳 整数平方根初始估计
整数 sqrt(x) 的牛顿迭代初始值可用 1 << ((32 - nlz(x)) / 2) 给出，比从1开始迭代收敛快约 log₂(x)/2 倍。
🔐 密码学：大数比较与模幂
RSA/ECC 大数模幂运算的循环次数 = 指数的位长 = 64 - nlz(e)。OpenSSL 的 BN_num_bits() 底层就是 CLZ。快1位 = 少1次模乘 = 节省数十微秒。
§5-4 尾随零计数 — Counting Trailing Zeros (ntz / CTZ)
// ntz(x) = 最低1位之后有多少个0 = 最低1位的位置 ntz(x) 等价于 x & (-x)（隔离最低1位）的位位置， 也等价于 x 能被 2^k 整除的最大 k。 Warren 给出了比 nlz 更多样的实现路径，其中最精妙的是 De Bruijn 乘法查表：用一个32位 De Bruijn 序列的特殊乘法， 将"最低1位的位置"唯一映射到一个5位索引，只需1次乘法+1次表查找。 ntz 的"尺子函数"（ruler function）应用尤为有趣： ntz(k) 描述了汉明编码中下一个要移动的磁盘，以及格雷码生成中下一个要翻转的位。

柱高 = ntz(k)，规律: 每隔1步插入0，每隔2步插入1，每隔4步插入2... → 汉诺塔第k步移动磁盘 ntz(k)
ntz 实现 — 多种方法
// §5-4 ntz 方法总览 // 方法1: 利用 nlz (先隔离最低1位) int ntz1(unsigned x) { return nlz(~x & (x-1)); // ~x & (x-1): 将最低1位及以下全部取反再清零 // 等价于将 x 的尾随零变成前导零 } // 方法2: x & (-x) 隔离最低1位 + nlz int ntz2(unsigned x) { if (x == 0) return 32; return 31 - nlz(x & -(int)x); // x & (-x): 只保留最低1位 // 31 - nlz: 最低1位的位置即尾随零数 } // 方法3: 二分搜索 (仿照 nlz) int ntz_bisect(unsigned x) { int n = 1; if (x == 0) return 32; if ((x & 0x0000FFFF)==0) { n+=16; x>>=16; } if ((x & 0x000000FF)==0) { n+= 8; x>>= 8; } if ((x & 0x0000000F)==0) { n+= 4; x>>= 4; } if ((x & 0x00000003)==0) { n+= 2; x>>= 2; } return n - (x & 1); } // 方法4: 硬件 // __builtin_ctz(x) (x≠0, 否则UB) // x86: BSF rd, rs (Bit Scan Forward) // x86 BMI1: TZCNT rd, rs (x=0时返回32) // ARM64: RBIT + CLZ (反转位再数前导零) // RISC-V B: CTZ rd, rs // C++20: std::countr_zero(x)
De Bruijn 序列查表 ntz — 最精妙实现
// §5-4 Warren Figure 5-22: De Bruijn ntz // 核心思想: De Bruijn 序列的每个连续子串唯一 // 0x077CB531 是一个32位 De Bruijn 序列常数 // 将 (x & -x) 乘以它, 取高5位 → 唯一映射到0~31 static const int deBruijn_ntz[32] = { 0, 1, 28, 2, 29, 14, 24, 3, 30, 22, 20, 15, 25, 17, 4, 8, 31, 27, 13, 23, 21, 19, 16, 7, 26, 12, 18, 6, 11, 5, 10, 9 }; int ntz_debruijn(unsigned v) { return deBruijn_ntz[ ((v & -(int)v) * 0x077CB531u) >> 27 ]; } // 步骤解析 (以 v=0x00000100 为例): // 1. v & -v = 0x00000100 (只留最低1位) // 2. * 0x077CB531 // = 0x077CB531 << 8 = 0xCB531000(低32位) // 3. >> 27 = 0xCB531000 >> 27 = 0x19 = 25 // 4. deBruijn_ntz[25] = 8 ✓ (0x100 = bit 8) // 为什么叫 De Bruijn? // 0x077CB531 的32位二进制展开是一个 De Bruijn // 序列 B(2,5): 每个5位子串 00000~11111 恰好 // 出现一次。乘以 2^k (= 左移k位) 恰好将第k个 // 5位子串移到高5位, 使其成为唯一的表索引。 // 只需32项表, 1次乘法, 1次移位, 1次查表!

§CRIT 深度点评 — 第5章的精华与启示

第5章：六十年智慧的结晶，从HAKMEM到C++20

✦ 分治 popcount 是"并行思维"的教科书

Warren §5-1 的分治方法的本质是把32个串行加法器在一个寄存器内同时并行运行。 这个思路直接对应了 Kogge-Stone 并行前缀加法器电路。 软件工程师习惯了"循环+计数器"的思维定势，而这个算法告诉你： 只要问题具有位级独立性，就可以在单个 ALU 周期内完成原本需要32步的工作。 这种"数据并行"思想是 SIMD、GPU Warp、TPU 矩阵单元的共同祖先。

✦ De Bruijn 查表法：数学与工程的完美接吻

De Bruijn 序列原本是组合数学中的纯理论对象（1940年代荷兰数学家 Nicolaas de Bruijn 研究的）， 六十年后被 Robert Harley（1996年）发现可以用来解决 nlz/ntz 的高效实现问题。 一次乘法 + 一次移位 + 一次表查找，不需要任何分支，不需要任何硬件扩展。 这种"把纯数学理论直接变成工程指令序列"的惊喜感，正是这本书最令人着迷的地方。

✦ 浮点借用法：跨域借鸡下蛋的极客精神

用 IEEE 754 double 的指数字段来计算整数对数，是一种令强迫症程序员既爱又恨的技巧。 它的正确性依赖于一个深刻的观察：浮点数的规范化指数本质上就是"最高有效位的位置"。 但其代价是：代码依赖未定义行为（type-punning），跨 ISA 的大小端问题， 以及整数和浮点寄存器之间昂贵的数据转移。 Warren 用整整一页讨论其"不稳定性"——这本身就是对编程实践诚实性的一种示范。

✦ 尺子函数与汉诺塔：意想不到的跨学科统一

ntz(k) = k 中2的最高次幂因子 = "尺子函数"，描述了一把标准尺子上各刻度线的高度。 Warren 指出它同时解决了三个看似毫无关联的问题：汉诺塔移盘顺序、格雷码翻转位、Gosper 环路检测算法。 这种"一个函数，多种解释"的多态性，是计算机科学中最美的现象之一。 有多少工程师在写汉诺塔递归时，从未意识到自己的递归调用栈就是一棵 ntz 的决策树？

⚠ 时代局限：硬件追上来了，但书的价值没有消失

x86 POPCNT（2008）、LZCNT/TZCNT（2013）、C++20 <bit>（2020）、 RISC-V B 扩展（2021）的相继标准化，让本章大部分"软件技巧"变成了博物馆展品。 但这恰恰说明：Warren 这本书所记录的，是一代工程师在硬件缺位时用智慧填补空白的历史。 理解这些技巧的原理，不是为了今天还用它们，而是为了理解：当你面对一个没有现成指令的新问题时， 应该怎样思考。这种"用现有原语组合出新原语"的能力，才是本章真正的遗产。
